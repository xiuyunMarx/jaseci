- **Fix: `jac check` failed on generated client code that wraps npm imports**: String-form client imports (`cl import from "clsx"`) are now resolved against the project's installed `.jac/client/node_modules` with Node-faithful resolution (`exports` maps with conditions and wildcard subpaths, `module`/`main` fallbacks, and scoped/deep specifiers), so installed packages no longer report `W1100 Module not found`. jac-shadcn's own generated `cn()` helper now type-checks cleanly. This is type-check only; JS bundling still leaves the bare specifier for vite to resolve.
- **Feature: npm imports are typed from their TypeScript declarations**: The checker now locates a package's `.d.ts` (package.json `types`/`typings`, the exports `types` condition, sibling declaration files, and `@types/*` fallback) and ingests a pragmatic subset (functions, consts, type aliases, unions, arrays, literals, optional/rest params), so `cl import from "clsx" { clsx }` gets its real declared signature; misusing a typed npm import is a genuine type error. Constructs outside the subset degrade to a declared foreign `any`.
- **Feature: gradual typing for untyped foreign imports + strict opt-out**: Values imported from an external module without declarations carry a *foreign* `any` that flows gradually (callable, assignable at strict sites) like untyped JS in TypeScript, reported as `W1102`; user-written `any` stays strict. Projects can set `[check] untyped-external = "error"` in jac.toml to escalate undeclared external imports to the new `E1120`.
- **Fix: missing-module Unknowns no longer cascade**: An import that already reported `W1100`/`W1101` produces a diagnosed Unknown that is suppressed downstream instead of re-escalating into hard `E1002`/`E1001`/`E1032` errors at every use site: one actionable diagnostic per root cause.
