- **Fix: the `jac` binary no longer leaks `PYTHONHOME`/`PYTHONPATH` into subprocesses**: the launcher configured its embedded CPython by exporting `PYTHONHOME`/`PYTHONPATH` (plus `PYTHONUTF8`/`PYTHONIOENCODING`/`PYTHONDONTWRITEBYTECODE`/`PYTHONNOUSERSITE`) into the process environment, so every child a jac program spawned inherited them; any python-based child (`aws` EKS exec-credentials, surfacing as `(401) Unauthorized` at database provisioning, GKE `gcloud`, plain `python3`) adopted the jac runtime as its home, could not find its own stdlib, and died with `Fatal Python error: Failed to import encodings module`. The launcher now hands all interpreter configuration to CPython directly via the PEP 741 init-config API (`Py_InitializeFromInitConfig`), so the `PYTHON*` variables never exist in the environment: children see exactly the environment `jac` was launched with (including the user's own `PYTHONPATH`, which was previously clobbered), and ambient `PYTHON*` shell variables can no longer perturb the bundled runtime (`use_environment=0`). The per-call-site strips in `kubernetes_realizer.jac`, `kubernetes_utils.impl.jac`, and the desktop `_dev_launch.jac` are removed as redundant.
